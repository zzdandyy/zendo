# i18n 国际化技术方案

## 现状

| 维度 | 情况 |
|------|------|
| 翻译基础设施 | 无 — 没有 i18n 库、没有 locale 目录 |
| 当前语言 | 全英文，无中文字符 |
| UI 字符串分布 | ~300 条，硬编码在 67 个 tsx 文件和 stores 中 |
| 字符串密度 Top 5 | SettingsPage(~90), HostEditModal(~20), PortForwardingPage(~16), HostsDashboard(~11), ExplorerView(~10) |

## 目标

- 中英文双语切换（留好扩展更多语言的空间）
- 语言偏好持久化到现有 SQLite settings 中
- 翻译文件按功能模块拆分，便于维护
- TypeScript 类型安全 — 拼写错误的 key 在编译时就能发现
- 不破坏现有功能，增量迁移

---

## 技术选型

### 为什么用 react-i18next

| 候选 | 结论 |
|------|------|
| **react-i18next** | ✅ 选它。React 生态事实标准，TypeScript 支持好，ICU 格式内置，文档和社区成熟 |
| react-intl (FormatJS) | ❌ 太重，API 偏繁琐，本项目体量不匹配 |
| 自研 Context | ❌ 造轮子，复数/插值/格式化都要手写 |
| lingui | ❌ 需要额外的 CLI 提取工作流，侵入性太强 |

### 依赖

```bash
pnpm add i18next react-i18next i18next-icu
```

- `i18next` — 核心运行时
- `react-i18next` — React 绑定（`useTranslation` hook）
- `i18next-icu` — ICU message 格式支持（复数、插值）

│   │   │   ├── common.json   # 通用：按钮、确认、取消、提示等
│   │   │   ├── settings.json # 设置页
│   │   │   ├── hosts.json    # 主机管理
│   │   │   ├── terminal.json # 终端
│   │   │   ├── sftp.json     # SFTP 文件管理
│   │   │   ├── s3.json       # S3
│   │   │   ├── snippets.json # 代码片段
│   │   │   ├── portforward.json

│   │       └── ...

### 命名空间（namespace）划分原则

按**功能域**划分，与 Zustand stores 的领域对齐：

| Namespace | 覆盖范围 |
|-----------|---------|
| `common` | 通用 UI：确认/取消/删除/保存/关闭/搜索/空状态 等 |
| `settings` | 设置页全部文案 |
| `hosts` | 主机列表、编辑、导入 |
| `terminal` | 终端相关：连接状态、断开覆盖层、搜索栏、窗格头 |
| `sftp` | SFTP 文件浏览器、工具栏、属性面板 |
| `s3` | S3 连接、浏览器 |
| `snippets` | 代码片段管理 |
| `portforward` | 端口转发 |
| `history` | 历史记录 |
| `transfers` | 文件传输进度 |

### Key 命名规范

层级点号分隔，从通用到具体：

```
common:
  button.save          → "Save" / "保存"
  button.cancel        → "Cancel" / "取消"
  status.loading       → "Loading..." / "加载中..."

settings.general:
  language.label       → "Language" / "语言"
  theme.label          → "Theme" / "主题"

terminal.pane:
  disconnect.title     → "Disconnected" / "已断开"
  reconnect.button     → "Reconnect" / "重新连接"
```

命名规则：
- 小写字母 + 点号分隔
- 动词在后的英文语序（`button.save` 而非 `save.button`）
- 嵌套深度 ≤ 4 层

### 语言检测器

不依赖 URL 或浏览器 `navigator.language`（这是桌面 app），而是：

```
                    ┌─────────────┐
首次启动 ──────────→│ 默认 en      │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
用户在设置页切换语言 │ settings-store │
                    │ lang: zh     │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │ i18next      │
                    │ changeLanguage│
                    │ → 所有组件   │
                    │   re-render  │
                    └─────────────┘
```

实现：
- `settings-store.ts` 已有 `settings` 对象，新增 `lang: 'en' | 'zh'` 字段
- `src/i18n/settings.ts` 创建一个自定义 i18next language detector plugin，从 Zustand store 读取初始语言
- 用户切换时，store 更新 → 写入 SQLite → 调用 `i18n.changeLanguage()`

### 类型安全

从英文翻译文件自动推导类型，这样 `t('key.typo')` 在 TS 编译时就报错：

```ts
// src/i18n/types.ts — 从 en/common.json 等文件推导
import common from './locales/en/common.json'
import settings from './locales/en/settings.json'
// ... 其他 namespace

// 递归解析嵌套 key 为点号路径
type DotPrefix<T, K extends string> = K extends string
  ? `${K}.${StringKey<T>}` : never

type StringKey<T> = {
  [K in keyof T & string]: T[K] extends string
    ? K
    : T[K] extends object
      ? DotPrefix<T[K], K>
      : never
}[keyof T & string]

type CommonKey = StringKey<typeof common>     // "button.save" | "button.cancel" | ...
type SettingsKey = StringKey<typeof settings>  // ...

// 重载 t 函数
declare module 'i18next' {
  interface CustomTypeOptions {
    resources: {
      common: typeof common
      settings: typeof settings
      // ...
    }
  }
}
```

这样 `t('button.sve')` → 编译报错，开发体验好。

## 分阶段实施计划

### Phase 1：基础设施 + 设置页试点（1-2 天）

工作量最小，验证整套流水线是否能跑通。

1. 安装 `i18next` / `react-i18next` / `i18next-icu`
2. 创建 `src/i18n/` 目录结构
3. 实现 `index.ts`（初始化）和 `settings.ts`（语言检测器）
4. 在 `App.tsx` 中挂载 `I18nextProvider`
5. 在 `settings-store.ts` 中新增 `lang` 字段
6. 创建 `locales/en/common.json` 和 `locales/zh/common.json`（只放 ~20 条通用文本）
7. **试点迁移 SettingsPage.tsx**（~90 个字符串，最复杂的页面）
8. 在设置页加入语言切换控件
9. 验证：切换语言 → 设置页即时翻译 → 关掉重开语言保持

> 选 SettingsPage 试点是因为它字符串最密集，如果这套方案能搞定 SettingsPage，其他页面不在话下。

### Phase 2：核心模块迁移（2-3 天）

按功能域逐块迁移，每块完成后测试验证：

| 顺序 | 模块 | 预估字符串量 |
|------|------|-------------|
| 1 | `common` — 把各组件中重复的通用文案抽到 common | ~30 |
| 2 | `hosts` — HostEditModal, HostsDashboard, HostCard, ImportSshConfigModal, GroupModal | ~50 |
| 3 | `terminal` — PaneHeader, DisconnectOverlay, TerminalSearchBar, TerminalArea | ~30 |
| 4 | `sftp` — ExplorerView, ExplorerToolbar, FilePropertiesDialog, PathBar | ~40 |
| 5 | `s3` — S3ConnectDialog, S3Page, S3Browser | ~20 |
| 6 | `snippets` — 全部 snippet 组件 | ~25 |
| 7 | `portforward` — PortForwardingPage | ~16 |
| 8 | `history` — HistoryPage | ~10 |
| 9 | `transfers` — TransferPopover | ~5 |

### Phase 3：Store + 边缘文案（1 天）

Zustand stores 中的 toast 消息、状态文案（`health-store`, `updater-store`, `port-forward-store` 等）。

这部分特殊：字符串在 store 里，不在组件里。两种做法：

- **方案 A**：在 store 里只用 key，由组件消费时翻译（推荐）
- **方案 B**：在 store 里调用 `i18n.t()` —— 需要 i18n 实例可被非 React 上下文访问

建议用方案 A：store 只暴露 key，翻译在 UI 层完成。

### Phase 4：Stale string 检查 + 文档（0.5 天）

- 写个简单脚本检查哪些 key 在代码中未被引用（通过 grep `t('`模式）
- 在 CLAUDE.md 中补充 i18n 开发规范

## 关键设计决策

### 1. 英文作为 key，还是用抽象 key？

| 方案 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| **英文作 key（推荐）** | `t("button.save")` | 英文文件本身可读性好；缺翻译时 fallback 到 key 本身不会出洋相 | key 和翻译耦合 |
| 抽象 key | `t("BTN_SAVE")` | 改了英文文案不需要改 key | 缺翻译时显示 `BTN_SAVE` 很难看；需要额外映射 |

**选方案 A（英文作 key）**。这是 react-i18next 社区的主流建议，也适合本项目（当前全部是英文，天然可以作为 key）。

### 2. ICU 格式的使用范围

只在**需要参数化**的场景用 ICU，简单字符串直接用：

```json
// 简单字符串 — 直接用
{ "button.save": "Save" }

// 需要插值 — 用 ICU
{ "transfer.progress": "{count} files remaining" }
{ "host.count": "{count, plural, one {# host} other {# hosts}}" }
```

本项目文案以标签和短句为主，复数场景很少，ICU 是备用能力。

### 3. 翻译文件是否拆分 namespace？

**拆**。~300 个字符串如果放一个文件，很快就上千行。按功能域拆到 10 个文件，每个 30-80 条，清晰可控。

### 4. 中英文翻译文件的 key 必须一致吗？

不强制。英文文件中的 key 是权威源（也是 TS 类型推导的依据）。中文文件缺失某个 key 时，自动 fallback 到英文。

建议后续加一个 CI 检查：列出中文文件中缺失或多余的 key。

## 开发规范（后续补充到 CLAUDE.md）

1. **新增文案必须走 i18n** — 不再在 JSX 中硬编码英文
2. **Key 命名** — 统一用小写点号分隔，动词在后
3. **翻译文件编辑** — 中英文文件一起改，避免只改一方
4. **不要全量重写翻译文件** — JSON 格式化差异会让 diff 不可读，只改目标行
5. **通用文案放 common** — 按钮文案、提示语等不要在每个 namespace 重复定义

## 风险与边界

| 风险 | 应对 |
|------|------|
| 迁移过程中改漏字符串 | Phase 2 结束后全量 grep `"[A-Z]` 检查 tsx 中的残留 |
| 翻译文件变味（JSON 嵌套过深） | 硬规则：嵌套 ≤ 4 层，超过就拆 namespace |
| 性能 | react-i18next 对 ~300 条字符串体量无感，零性能问题 |
| 后端 Rust 错误信息 | 依然是英文。后端错误不在此次范围内（数量少且要改 Rust 代码）。如需，可在前端做个简单的错误码映射表 |
